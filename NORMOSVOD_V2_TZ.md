## Главный Архитектурный Принцип V2

Для документа должен существовать один источник правды и два представления:

- `screen-flow representation`
- `print-A4 representation`

## 1. Аудит Текущего Проекта

### 1.1 Что это за проект сейчас

Сейчас Normosvod — это статический сайт-каталог автономных HTML-viewer документов.

Текущая фактическая модель:

- каталог и главная страница работают от `public/data/documents.json`, `public/data/stats.json`, `public/data/search-index.json`;
- документная карточка доступна по маршруту `/doc/:slug`;
- сам документ публикуется отдельно как `/docs/:slug/viewer.html`;
- карточка документа не является новым reader-движком, а лишь оболочкой над legacy viewer;
- просмотр документа в основном сценарии фактически уводит пользователя в автономный viewer.

### 1.2 Как реально устроены маршруты

Проект собран как `Vite SPA`.

Подтверждено:

- `vite.config.mjs`: `appType: 'spa'`
- `src/js/app.js`: собственный client-side router
- `public/.htaccess`: все несуществующие пути отдаются в `index.html`

Реальные маршруты runtime:

- `/` — главная
- `/catalog` — каталог
- `/doc/:slug` — карточка документа

Сервисные файлы:

- `src/catalog.html` — meta refresh на `/catalog`
- `src/document.html` — meta refresh на `/`
- `src/404.html` — redirect-механика для SPA fallback

Вывод:

- проект не является набором реальных отдельных страниц каталога и документа;
- это один SPA shell с минимальной маршрутизацией;
- runtime документа внутри SPA отсутствует как самостоятельная система.

### 1.3 Как реально устроены viewer

Каждый viewer — это автономный HTML-файл со своим CSS, DOM и JS runtime.

Подтверждённые свойства по `public/docs/gost-29319-2025/viewer.html`:

- есть `.app`, `.sidebar`, `.workspace`, `.page`;
- есть `NAV_ITEMS`;
- есть локальный поиск по документу через `docSearchInput`;
- есть `themeToggleBtn`;
- есть `biblioBackBtn`;
- есть `@media print`;
- на экране показывается только одна `.page.active`;
- переход по документу работает через `goToPage()`, `navigateToHash()`, `navigateToElement()`.

Фактическая модель viewer:

- документ хранится как набор жёстко сверстанных A4-страниц;
- экранный режим — не потоковое чтение, а переключение между страницами;
- поиск внутри viewer page-based;
- quick-nav завязан на `NAV_ITEMS`;
- печать работает потому, что исходная DOM-модель уже page-based и A4-ориентированная.

Вывод:

- текущий viewer — это самостоятельный page-runtime;
- он не является подходящей основой для профессионального экранного инструмента;
- его ценность — в контенте, anchors, навигации и print-геометрии, а не в финальной архитектуре.

### 1.4 Как реально устроен import pipeline

Текущий import pipeline не строит структурированную документную модель.

Фактический pipeline:

1. `scripts/import-viewer.mjs` читает входной `viewer.html`
2. `validateViewerHtmlBasic()` проверяет наличие базовых viewer-структур
3. `extractViewerMeta()` извлекает:
   - `gostNumber`
   - `title`
   - `shortTitle`
   - `year`
   - `pages`
   - `tags`
   - флаги viewer-возможностей
   - `navItems`
4. `generatePreviewOrPlaceholder()` генерирует `preview.html`
5. рядом с документом пишется `meta.json`
6. пересобираются:
   - `public/data/documents.json`
   - `public/data/stats.json`
   - `public/data/search-index.json`

Фактические артефакты импорта:

- `/docs/:slug/viewer.html`
- `/docs/:slug/meta.json`
- `/docs/:slug/preview.html`
- `public/data/documents.json`
- `public/data/stats.json`
- `public/data/search-index.json`

Вывод:

- текущий pipeline — упаковщик и индексатор legacy-viewer;
- он не создаёт canonical document model;
- он не делит документ на смысловые блоки;
- он не строит сущности, связи, определения, тематические режимы, слой связанных норм.

### 1.5 Как реально устроен поиск

Поиск в каталоге построен по page-based индексу.

Текущая схема:

- `scripts/lib/extract-search-text.mjs` извлекает из viewer список страниц;
- каждая запись индекса имеет:
  - `pageIndex`
  - `anchor`
  - `text`
- `src/js/search.js` объединяет metadata haystack и page text;
- ранжирование строится по вхождениям в title, manifest и page text;
- лучший hit возвращает `pageIndex`, `anchor`, `snippet`, `matchedPages`.

Следствия:

- поиск логически завязан на страницу, а не на смысловой блок;
- результат ведёт либо к карточке документа, либо к anchor внутри legacy viewer;
- поиск не умеет работать по типам блоков, сущностям, связанным нормам, определениям, предупреждениям.

### 1.6 Как реально устроен deploy

Deploy подтверждён по коду и workflow.

Текущая схема публикации:

- локальный компьютер
- GitHub
- GitHub Actions
- Reg.ru по SSH/SCP

Подтверждено:

- `.github/workflows/deploy-reg-ru.yml`
- `scripts/deploy-reg-ssh.mjs`
- `scripts/create-deploy-archive.mjs`
- `public/.htaccess`
- `vite.config.mjs`

Что реально уходит на сервер:

- `dist/assets`
- `dist/data`
- `dist/docs`
- `dist/index.html`
- `dist/404.html`
- `dist/.htaccess`

Вывод:

- проект уже имеет рабочий статический deploy contour;
- architectural rewrite не требует смены hosting-модели;
- backend для V2 не нужен.

### 1.7 Сильные стороны текущей архитектуры

- статический deploy уже работает;
- контент публикуется как независимые doc-artifacts;
- есть manifest, stats, search index;
- есть детерминированный import flow;
- есть базовая проверка входных viewer-файлов;
- есть архивирование заменённых и импортированных исходников;
- есть уже работающая печать A4;
- есть базовые тесты на HTML safety, rendering safety, search logic.

### 1.8 Ограничения, которые мешают сделать профессионального помощника

Критические ограничения:

- экранная модель документа page-based, а не semantic-flow;
- reader как единая система отсутствует;
- SPA не владеет содержимым документа, а только оборачивает legacy viewer;
- import pipeline не создаёт семантическую структуру;
- поиск не знает блоков, сущностей и связей;
- визуальная система каталога не масштабируется в document-platform;
- тематические режимы отсутствуют;
- документ на экране воспринимается как листовая верстка, а не как рабочий инструмент.

Жёсткий вывод:

текущую архитектуру нельзя превратить в V2 простой перекраской и мелкими UX-правками.

## 2. Что Сохранить / Что Заменить

### 2.1 Сохранить без изменений

- `Vite` как static build tool
- текущий static hosting contour
- GitHub Actions deploy на Reg.ru
- GitHub Pages workflow
- `public/.htaccess`
- директорию публикации `dist/docs`
- общую идею manifest-артефактов в `public/data`

### 2.2 Сохранить, но обернуть новым слоем

- `documents.json` как переходный каталоговый источник
- `meta.json` как legacy metadata source
- текущие `viewer.html` как legacy runtime
- `preview.html` как переходный preview artifact
- slug/hash/archive discipline из import pipeline

### 2.3 Переписать

- экранный reader
- карточку документа
- каталоговую информационную архитектуру
- поисковую модель
- data schema документа
- импортный pipeline
- систему тематических режимов
- визуальную систему платформы

### 2.4 Вывести в legacy

- embedded `iframe viewer` как основной сценарий
- current page switching runtime
- page-only internal search как главный способ чтения
- current card model, где пользователь фактически переходит в автономный viewer

### 2.5 Использовать только как контентный источник

- page HTML
- anchors
- `NAV_ITEMS`
- библиографические переходы
- current print geometry
- existing section ids

### 2.6 Жёсткое решение

Сохранять legacy viewer как целевую архитектуру запрещено.

Допустим только такой статус:

- `legacy artifact`
- `migration source`
- `fallback mode`

## 3. Продуктовая Концепция V2

### 3.1 Новая продуктовая модель

Normosvod V2 — это не каталог HTML-viewer файлов.

Normosvod V2 — это профессиональный экранный помощник по нормативной документации.

### 3.2 Целевая аудитория

- технологи и специалисты по ЛКМ
- инженеры и проектировщики
- специалисты по строительным и инженерным нормам
- специалисты по пожарной безопасности
- специалисты по техническому регулированию и применению нормативных документов
- эксперты по соответствию и внутреннему контролю

### 3.3 Основной пользовательский сценарий

1. пользователь находит документ или норму через каталог или поиск
2. открывает не legacy viewer, а профессиональный reader
3. читает документ как экранную смысловую ленту
4. быстро видит:
   - требования
   - определения
   - обязательные условия
   - исключения
   - связанные нормы
   - приложения
   - предупреждения
5. при необходимости переходит:
   - к связанному блоку
   - к связанной норме
   - к A4 print view
   - в legacy mode

### 3.4 Как должен восприниматься документ на экране

Документ должен восприниматься как структурированный рабочий объект, а не как пачка листов.

Экранное чтение должно быть организовано как безразрывный поток внутри рабочего пространства reader.

Ключевое требование:

- пользователь не должен воспринимать экранный режим как набор разорванных карточек или отдельных листов;
- документ должен читаться сверху вниз как непрерывная лента;
- левая панель структуры и правая contextual rail должны сопровождать поток чтения, а не ломать его;
- A4-пагинация допускается только как print-layer и locator-layer, но не как основная модель экранного чтения.

На экране должны появиться типизированные блоки:

- раздел
- подраздел
- определение
- нормативная ссылка
- обязательное требование
- условие применения
- предупреждение
- примечание
- формула
- таблица
- приложение
- библиографическая ссылка
- редакционные сведения

### 3.5 Как должны работать связи и подсказки

Обязательные механики:

- `tooltip` для определений и терминов
- `related norms` для нормативных ссылок
- `context cards` для связанных документов
- `warning blocks` для обязательных и ограничительных положений
- `practice accents` для блоков, важных в рабочем применении
- быстрый переход к связанному блоку внутри документа
- быстрый переход в соседний документ по сущности или ссылке

### 3.6 Как должен выглядеть каталог V2

Каталог V2 — это не просто сетка карточек.

Он должен уметь:

- фильтровать по тематике;
- фильтровать по типу документа;
- фильтровать по статусу;
- фильтровать по году;
- фильтровать по наличию определений, таблиц, приложений;
- показывать не только title, но и краткий рабочий synopsis;
- выводить релевантные блоки, а не только документ целиком.

### 3.7 Как должна выглядеть карточка документа V2

Карточка документа должна показывать:

- номер и наименование документа
- тему и тематический режим
- тип документа
- статус
- год
- область применения
- ключевые разделы
- связанные нормы
- основные определения
- способ чтения:
  - `Читать в V2`
  - `Открыть печатную версию`
  - `Открыть legacy viewer`

### 3.8 Как должен выглядеть reader V2

Reader должен состоять из трёх функциональных зон:

- левая панель структуры документа
- центральная зона потокового чтения
- правая contextual rail зона

Reader должен занимать всю ширину окна браузера как рабочее приложение, а не как узкий центрированный лендинг.

Требование к компоновке:

- основная shell-композиция должна использовать full-width layout;
- левая панель структуры и правая contextual rail должны быть частью общей рабочей ширины страницы;
- центральная колонка чтения может иметь собственную оптимальную ширину текста, но вся платформа не должна быть ограничена узким контейнером сайта;
- на больших экранах интерфейс должен восприниматься как нормативная рабочая среда, а не как обычная маркетинговая страница.

Правая contextual rail должна показывать:

- связанные нормы
- определения
- предупреждения
- быстрые переходы
- данные о печатной странице текущего блока

## 4. Техническая Архитектура V2

### 4.1 Платформа

Решение:

- `Vite` оставить
- `TypeScript` ввести
- перейти на компонентный UI runtime

Рекомендуемый стек:

- `Vite`
- `TypeScript`
- `Preact`

Причина выбора:

- проект остаётся статическим;
- нет необходимости в backend;
- нужен управляемый stateful reader;
- нужна строгая типизация документной схемы;
- нужен компонентный подход для reader shell, block renderer, contextual rail, search UI.

### 4.2 Каноническая модель данных

Единый источник правды:

- canonical structured document package

Рекомендуемое размещение:

- `content/docs/<slug>/document.json`

В canonical document должны храниться:

- `documentMeta`
- `themeId`
- `docType`
- `status`
- `blocks[]`
- `entities[]`
- `relations[]`
- `printMap[]`
- `legacy`

### 4.3 Формат документа

Решение:

- не HTML-only;
- не page-array-only;
- не flat JSON;
- использовать гибридный `JSON AST`.

Структура:

- block tree для screen renderer
- inline fragments для ссылок, терминов и врезок
- relation graph для связанных норм и сущностей
- print hints для A4 renderer
- legacy references для обратной совместимости

### 4.4 Разделение screen и print

Нужно жёсткое разделение двух renderer-слоёв.

#### Screen renderer

- строится из canonical block model
- отображает infinite reading flow
- знает блоки, связи, tooltip, акценты, rail context
- работает в full-width application shell
- не должен воспроизводить page-frame поведение legacy viewer даже визуально

#### Print renderer

- строится из того же canonical source
- генерирует отдельный A4-ориентированный output
- обеспечивает стабильную печать

Запрещённый вариант:

- использовать один и тот же page-based HTML как базовую архитектуру для обоих режимов.

### 4.5 Как сохранить A4 pagination

Печать должна остаться A4-корректной.

Решение:

- A4 не хранить как главный экранный runtime;
- A4 генерировать как отдельное представление из canonical source;
- в canonical schema хранить print hints;
- хранить связь блока с печатной страницей через `printMap`.

Обязательные поля print-слоя:

- `pageNumber`
- `pageAnchor`
- `blockIds`
- `keepWithNext`
- `avoidBreakInside`
- `tableSplitPolicy`

### 4.6 Как мигрировать существующие viewer

Текущий `viewer.html` должен стать источником миграции.

Миграционный parser должен:

1. прочитать page DOM
2. извлечь `NAV_ITEMS`
3. извлечь anchors
4. извлечь page text
5. попытаться сегментировать документ в semantic blocks
6. извлечь references и candidate entities
7. сохранить unmapped content как `raw-html blocks`

### 4.7 Новый import pipeline

Новый pipeline:

1. ingest legacy viewer
2. validate structure
3. extract metadata
4. extract pages and anchors
5. segment semantic blocks
6. extract entities and references
7. assign theme
8. write canonical `document.json`
9. generate screen data
10. generate print representation
11. generate search indices
12. update manifest

### 4.8 Поиск V2

Поиск должен стать трёхслойным.

#### Уровень 1. Document index

Поиск по:

- номеру документа
- названию
- короткому synopsis
- тематике
- тегам

#### Уровень 2. Block index

Поиск по:

- заголовкам блоков
- телу требований
- определениям
- примечаниям
- приложениям

#### Уровень 3. Entity / relation index

Поиск по:

- нормативным ссылкам
- терминам
- методам
- ГОСТ/ФЗ/ISO/ASTM сущностям

Page-based index допускается сохранить только как:

- `legacy search support`
- `print locator support`

### 4.9 Тематические стили

Тематические режимы должны быть частью data model, а не только CSS.

В manifest/document schema должно быть поле:

- `themeId`

Обязательные темы:

- `coatings`
- `regulation`
- `fire-safety`
- `construction`

Экранные режимы:

- выразительные;
- деловые;
- не кислотные.

Печатные режимы:

- приглушённые;
- контрастные;
- без визуального шума.

## 5. Полноценное ТЗ На Переработку

### 5.1 Цель

Перевести Normosvod из статического каталога автономных HTML-viewer документов в статическую профессиональную платформу экранной работы с нормативной документацией при сохранении существующего deploy-контура.

### 5.2 Границы проекта

Входит в проект:

- новый каталог V2
- новая карточка документа
- новый screen reader
- новый print renderer
- новая data model
- новый import pipeline
- block/entity/relation search
- thematic modes
- legacy mode

Не входит в проект:

- backend
- авторизация
- персональные кабинеты
- совместное редактирование
- пользовательские комментарии
- облачное хранение документов вне репозитория

### 5.3 Роли пользователей

#### Специалист

Использует каталог, поиск, reader, print.

#### Контент-оператор

Импортирует новые документы, проверяет корректность миграции и тематической классификации.

#### Технический администратор

Поддерживает pipeline, CI/CD, артефакты публикации и целостность схемы.

### 5.4 Функциональные требования

Система должна:

- открывать документ в V2 reader;
- отображать структуру документа как semantic flow;
- поддерживать block anchors;
- показывать связанные нормы;
- показывать определения и пояснения через tooltip/context card;
- поддерживать быстрые переходы между блоками;
- поддерживать переход к print representation;
- поддерживать переход в legacy viewer;
- поддерживать тематические режимы;
- поддерживать поиск по документам, блокам и сущностям;
- поддерживать карточку документа с рабочим synopsis.

### 5.5 Нефункциональные требования

- проект должен оставаться статическим;
- сборка должна выполняться в CI без backend сервисов;
- deploy должен оставаться совместимым с Reg.ru и GitHub Pages;
- данные должны быть детерминированными и сериализуемыми;
- schema changes должны быть типизированы;
- рендеринг должен быть безопасным относительно HTML/URL injection;
- проект должен работать на desktop и mobile;
- ключевые ссылки должны сохранять deep-link behavior.

### 5.6 UX/UI требования

- интерфейс должен быть деловым и выразительным;
- документ на экране не должен выглядеть как монохромная листовая масса;
- должны быть чётко выделены уровни важности;
- все типы блоков должны иметь визуальную иерархию;
- у документа должен быть тематический визуальный режим;
- правая contextual rail должна давать рабочую ценность, а не декоративный шум;
- экранный режим должен быть flow-first, не page-first.
- сайт должен использовать full-width layout, а не узкий центрированный контейнер по всей платформе;
- визуальная система должна быть минималистичной, спокойной и профессиональной;
- допустимо ориентироваться по характеру на современные нормативные и справочные интерфейсы уровня приложенных референсов, но запрещено повторять чужой дизайн точь-в-точь;
- референс должен использоваться как ориентир по степени лаконичности, рабочей плотности и организации интерфейса, а не как источник прямого копирования;
- минимализм не должен превращаться в пустоту: интерфейс обязан сохранять читаемую иерархию, контраст и рабочие акценты.

### 5.7 Требования к reader

Reader должен:

- рендерить документ как бесконечную ленту;
- поддерживать sticky outline navigation;
- поддерживать contextual rail;
- выделять типы блоков;
- отображать связанность между нормами;
- отображать печатную страницу текущего блока;
- иметь стабильные anchors;
- поддерживать ссылку на любой блок;
- иметь fallback в legacy mode.
- обеспечивать безразрывное чтение документа в центральной зоне без переключения между экранными страницами;
- использовать full-width shell с рабочими боковыми панелями и непрерывной центральной областью чтения;
- сохранять ощущение цельного документа при прокрутке, даже если внутри есть блоковые акценты, таблицы и приложения;
- не имитировать листовой viewer как основную экранную механику.

### 5.8 Требования к печати

- печатная версия должна быть отдельным представлением;
- печать должна сохранять корректную A4-структуру;
- таблицы, формулы и приложения не должны разрушаться при печати;
- screen theme не должен агрессивно переноситься в print;
- print representation должен строиться из того же canonical source, что и screen representation.

### 5.9 Требования к поиску

Поиск должен:

- индексировать документы;
- индексировать блоки;
- индексировать сущности и ссылки;
- возвращать не только документ, но и релевантный блок;
- показывать snippet на уровне блока;
- уметь переводить пользователя к V2 block anchor;
- при необходимости показывать print page locator;
- при отсутствии V2 migration уметь переводить в legacy anchor.

### 5.10 Требования к импорту

Импорт должен:

- принимать существующие legacy viewer-файлы;
- валидировать их структуру;
- извлекать metadata;
- извлекать anchors, nav, page text;
- строить canonical document package;
- помечать документы статусом миграции;
- генерировать производные артефакты screen/print/search;
- не ломать текущую публикацию viewer-файлов.

### 5.11 Требования к данным

Обязательные поля manifest/document layer:

- `slug`
- `gostNumber`
- `title`
- `shortTitle`
- `docType`
- `themeId`
- `status`
- `year`
- `readerMode`
- `migrationStatus`
- `legacyViewerUrl`
- `printUrl`
- `blocks`
- `entities`
- `relations`
- `printMap`

### 5.12 Требования к деплою

- существующие GitHub Actions workflows должны сохраниться;
- существующий SSH/SCP deploy на Reg.ru должен сохраниться;
- V2 не должен требовать нового инфраструктурного контура;
- новые артефакты должны публиковаться через тот же `dist`;
- старые URL `docs/:slug/viewer.html` должны оставаться доступными.

### 5.13 Критерии приёмки

Работа считается принятой, если:

- документ может открываться в V2 без iframe;
- document screen mode работает как semantic flow;
- каждый V2-документ имеет печатное A4 представление;
- связанные нормы и определения доступны из reader UI;
- поиск возвращает block-level результаты;
- тематический режим документа определяется и применяется;
- legacy viewer остаётся доступным;
- текущий production deploy не сломан;
- unmigrated документы продолжают открываться в legacy mode.

## 6. Стратегия Миграции

### 6.1 Основной вариант

V2 должен разрабатываться внутри текущего репозитория параллельно legacy.

### 6.2 Почему не переписывать поверх текущего runtime

Потому что текущий runtime:

- page-based;
- не владеет содержимым документа как данными;
- завязан на iframe/автономный viewer;
- не имеет семантической модели.

### 6.3 Почему не выносить в отдельный проект

Потому что уже существуют:

- рабочий build;
- рабочий deploy;
- рабочая публикация doc artifacts;
- import discipline;
- manifest/search/statistics pipeline.

Вынос в отдельный проект создаст:

- дублирование CI/CD;
- дублирование content pipeline;
- дополнительный риск рассинхронизации.

### 6.4 Как не сломать текущий прод

До полного cutover должны сохраняться:

- текущие `viewer.html`
- текущие `meta.json`
- текущие `documents.json`
- текущие маршруты `/doc/:slug`
- прямые legacy URL `/docs/:slug/viewer.html`

### 6.5 Legacy mode

Нужен обязательный режим:

- `readerMode: legacy`

Он должен использоваться для:

- документов без V2-migration;
- документов с неполной block segmentation;
- документов, не прошедших print verification;
- аварийного fallback.

### 6.6 Переключение маршрутов

Маршрут `/doc/:slug` должен стать маршрутом входа в подходящий reader mode.

Правила:

- если `readerMode = v2`, открывать новый reader;
- если `readerMode = hybrid`, открывать V2 shell с возможностью fallback;
- если `readerMode = legacy`, открывать текущую карточку/legacy entry.

### 6.7 Переход от current viewer model к block-based model

Миграция должна идти по документам, а не big-bang релизом.

Статусы миграции:

- `imported`
- `segmented`
- `entity-linked`
- `print-verified`
- `v2-ready`

### 6.8 Использование existing docs

Существующие документы использовать как:

- источник текстового контента;
- источник anchors;
- источник page boundaries;
- источник структурных заголовков;
- источник legacy print layout;
- источник bibliographic links.

Использовать existing docs как финальную V2 архитектуру запрещено.

## 7. План Реализации

### Этап 0. Аудит и стабилизация

Задачи:

- зафиксировать текущие контракты manifest/search/deploy;
- расширить manifest schema полями миграции;
- добавить schema validation;
- усилить тесты import pipeline;
- подготовить базовую структуру V2 данных.

Результат:

- репозиторий стабилизирован;
- добавлены V2 metadata fields;
- migration layer определён.

### Этап 1. Данные и модель документа

Задачи:

- ввести canonical `document.json`;
- ввести типы блоков;
- ввести типы сущностей и связей;
- реализовать parser из legacy viewer;
- построить print map schema.

Результат:

- появляется единый источник правды для V2;
- legacy viewer перестаёт быть единственной формой хранения документа.

### Этап 2. Новый reader

Задачи:

- реализовать screen-flow renderer;
- реализовать left outline;
- реализовать contextual rail;
- реализовать block anchors;
- реализовать tooltip/context interactions;
- реализовать print locator.

Результат:

- появляется профессиональный экранный reader.

### Этап 3. Каталог и карточка документа

Задачи:

- переработать каталог под block/entity-aware выдачу;
- переработать карточку документа в рабочий synopsis;
- добавить режимы `Читать`, `Печать`, `Legacy`.

Результат:

- каталог и документная entry-point логически переходят к платформенной модели.

### Этап 4. Тематические режимы

Задачи:

- внедрить token system;
- реализовать темы:
  - coatings
  - regulation
  - fire-safety
  - construction
- развести screen и print palette behavior.

Результат:

- документы перестают выглядеть одинаково;
- тема становится частью UX и data model.

### Этап 5. Миграция документов

Задачи:

- прогнать текущие документы через parser;
- довести блоковую разметку до `v2-ready`;
- проверить entity extraction;
- проверить print correctness;
- перевести import новых документов на V2-first pipeline.

Результат:

- контент начинает жить в новой модели;
- legacy становится вторичным слоем.

### Этап 6. Деплой и cutover

Задачи:

- опубликовать V2 параллельно legacy;
- включить маршрутное переключение по `readerMode`;
- оставить legacy URL рабочими;
- перевести `v2-ready` документы на новый reader;
- выполнить поэтапный cutover.

Результат:

- production не ломается;
- V2 вводится поэтапно;
- legacy сохраняется как fallback.

## Финальная Архитектурная Рекомендация

Основной вариант для проекта:

**2) новый V2 внутри текущего репозитория параллельно legacy**

### Обоснование

#### Почему не вариант 1

Глубокая переработка поверх текущего runtime ошибочна, потому что:

- текущий runtime page-based;
- текущий reader не владеет документной семантикой;
- SPA не контролирует документ как данные;
- архитектура текущего viewer не подходит под flow-based экранный инструмент.

#### Почему не вариант 3

Новый отдельный проект избыточен, потому что:

- deploy уже рабочий;
- import pipeline уже рабочий;
- публикация doc-artifacts уже рабочая;
- контентный и инфраструктурный контур уже существует;
- разделение создаст дублирование и риск рассинхронизации.

#### Почему вариант 2 лучший

Он даёт:

- greenfield reader и data architecture;
- сохранение текущего deploy и URL contracts;
- плавную миграцию по документам;
- возможность держать legacy как fallback;
- минимальный инфраструктурный риск;
- максимальный контроль над cutover.

### Итоговое решение

Normosvod V2 должен строиться в этом же репозитории как параллельная платформа с:

- новым canonical document model;
- двумя renderer-представлениями;
- legacy mode;
- поэтапной миграцией документов;
- сохранением текущего production deploy-контура.
